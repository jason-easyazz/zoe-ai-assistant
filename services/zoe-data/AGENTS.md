# services/zoe-data/ ‚Äî production web + chat API

## Purpose

THE production Zoe API: FastAPI app served by host uvicorn on port 8000. Owns chat, intent routing, memory, MCP tools, auth integration, background jobs, and the OpenClaw/Hermes agent bridges.

## Ownership

- App entry and core: `main.py`, `database.py` (+ `db_pool.py`, `db_compat.py`, `alembic/` migrations), `auth.py`.
- Routing/NLU: `intent_router.py`, `intent_classifier_llm.py`, `pi_intent_classifier.py`, `routers/` (see child doc).
- Agents and tools: `mcp_server.py`, `openclaw_ws.py`, `hermes_http.py`, `zoe_agent`-related modules, `executors/` (engineering-board executors, currently `kanban_adapter.py`).
- Agent security helpers: `agent_safety.py` ‚Äî the single home for the bash-tool shell-injection guard (argv parsing, no shell) and the SSRF guards: `assert_public_url` (web fetch), `assert_panel_host` (panels = private-LAN only; loopback/link-local/metadata/public always rejected, allowlist narrows but never bypasses), `guarded_urlopen` (validates + **pins the connection to the validated IP**, defeating DNS-rebinding), and `guard_browser_page` (Playwright `page.route` that validates every request/redirect hop pre-connect). Stdlib-only so it loads in slim CI.
- Memory: `hindsight_memory.py`, `hindsight_retain_candidates.py`, `conversation_context.py`.
- Memory ops: Ingest (`memory_digest.py` dreaming), Query (`memory_service.py` search), and Lint (`memory_lint.py`) ‚Äî the report-only scan for contradictions/stale/orphan/duplicate rows. The `/api/memories/for-prompt` recall packet (`routers/memories.py`, shared by chat + voice) selects bullets by relevance but PRESENTS conflicting bullets (same fact, changed value ‚Äî ‚â•0.6 token overlap, non-subset) newest-first by `added_at`, so a user correction always appears above its stale sibling; no-conflict packets are byte-for-byte unchanged (`test_memory_for_prompt.py`).
- Seam recall floor: `zoe_flue_client.py` ‚Äî flag-gated (`ZOE_SEAM_RECALL_INJECT`, default OFF, per-call env read) deterministic injection of the `/api/memories/for-prompt` packet into the outbound flue message on conservative personal-question turns ("what's my‚Ä¶", "do you remember‚Ä¶"), fetched in-process via `routers.memories.memory_for_prompt`, capped (12 bullets / 1600 chars), placed AFTER the ` zoe-uid:` identity line so the sidecar's single-line strip regex is untouched. A floor under the ~97% recall_memory tool-invocation ceiling, not a replacement for the tool. Fail-open: any fetch failure continues the turn without the block. Operator enables via env only after the replay gate. Pinned by `tests/test_flue_seam_recall_inject.py`.
- Relationship graph: `relationship_graph.py` ‚Äî READ-ONLY, flag-gated (`ZOE_RELATIONSHIP_GRAPH_ENABLED`, default OFF) bounded multi-hop traversal over `person_relationships` (portable recursive CTE, owner-scoped, cycle-safe, depth‚â§4/limit‚â§200). Explicit-query only (`GET /api/people/{id}/graph`), OFF the chat hot path. ADR: `docs/adr/ADR-relationship-memory.md`. **Graph-adjacency recall boost** (`memory_service._semantic_search`, design `docs/architecture/gbrain-memory-graph-recall.md` Part B): a 7th additive blend term that lifts facts stored under people 1‚Äì2 hops from the person a recall query names (surfacing "my sister's husband's job" when the fact lives on the husband node). Gated behind BOTH `ZOE_RELATIONSHIP_GRAPH_ENABLED` AND a new sub-flag `ZOE_GRAPH_RECALL_BOOST` (default OFF); weight `ZOE_GRAPH_RECALL_WEIGHT` (default 0.30). The neighbourhood is fetched best-effort on the ASYNC side (`search` ‚Üí `_graph_depth_by_pid`, reusing `person_extractor._resolve_person_uuid` + `relationship_graph.neighbors`, no new NLU) and passed as a plain `{people.id: depth}` dict into the sync blend; any failure ‚Üí empty dict ‚Üí no boost. OFF is byte-identical to pre-2b (term neither computed nor added, zero DB work). Proven by `tests/test_graph_recall_boost_eval.py` (multi-hop P@5/R@5 ON-vs-OFF + no-regression). Enable in prod only after the W0 RAM/replay gate on `ZOE_RELATIONSHIP_GRAPH_ENABLED`.
- Person merge / entity resolution: `person_merge.py` ‚Äî flag-gated (`ZOE_PERSON_MERGE_ENABLED`, default OFF) fold of an `is_partial` stub into a real contact, re-pointing all its data (satellite tables + `people.introduced_by_person_id` self-FK + `person_relationships`). Resolves the two conflicts a re-point creates under the partial current-edge index: drops self-edges (a==b) and de-dups colliding CURRENT edges (pre-checked before UPDATE so `person_relationships_pair_active` is never violated); target-wins field merge, source soft-deleted. Explicit user-triggered endpoint (`POST /api/people/{source_id}/merge-into/{target_id}`), owner-scoped, OFF the chat hot path (no voice replay gate). Roadmap item 4 of `docs/adr/ADR-relationship-memory.md`.
- Contacts from known people: the `person_create` pending-suggestion action (`pending_suggestions.py`, flag `ZOE_PERSON_SUGGEST_ENABLED`, default OFF) turns a confirmed proposal into a full editable `people` row (deduped, `_looks_like_person_name`-guarded, private by default). `contact_backfill.py` (`backfill_contacts`, flag `ZOE_CONTACT_BACKFILL_ENABLED`, default OFF) is the one-shot admin pass that feeds it: reads the user's person knowledge from BOTH the narrative layer ‚Äî the user's OWN `person`/`fact`/`relationship` MemPalace memories (family-shared rows owned by another user are excluded, so a household member's people don't leak in) AND the synthesized third-person `user_portraits` prose ‚Äî then extracts distinct name (+relationship) with two complementary passes (deterministic regex borrowed from `person_extractor` PLUS an LLM extraction over the combined text, reusing the local Gemma client idiom; malformed/failed LLM response falls back to regex-only), dedups against existing contacts, and emits a batch of `person_create` proposals ‚Äî never a silent contact write. Internal/loopback endpoint `POST /api/memories/backfill-contacts` (`require_internal_token`); pass `session_id` = the user's ACTIVE session (suggestion retrieval filters by session, so the default `backfill` session just parks them for an out-of-band accept flow). Byte-for-byte no-op when the flag is off. ADR: `docs/adr/ADR-contacts-from-known-people.md`.

## Local Contracts

- CRITICAL FILES (never delete): `main.py`, `database.py`, `auth.py`, `openclaw_ws.py`, `openclaw_maintenance.py`, `intent_router.py`, `mcp_server.py`, `routers/chat.py`, `routers/system.py`.
- Runs as a systemd USER service: `systemctl --user restart zoe-data.service`; in scripts/CI prefix `XDG_RUNTIME_DIR=/run/user/$(id -u)`.
- Every memory write carries scope (`personal` / `shared` / `ambient`); no unscoped writes.
- `ambient_memory` rows always carry the owning `user_id` (migration `0017`): the `/api/voice/ambient` write path resolves the panel's user via `_resolve_panel_default_user` and SKIPS the insert when unresolvable (never store ownerless audio); the `ambient_search` MCP tool filters on the caller's resolved user unconditionally (`tests/test_ambient_memory_scoping.py`).
- Per-turn extraction mines the USER turn only ‚Äî never feed the assistant reply (or a combined transcript) into `memory_extractor`, `person_extractor`, or `person_extractor_llm` as fact-source text. Zoe's own sentences stored as user facts poison the recall packet (live bug 2026-07-07: "‚Ä¶but I don't have a specific favorite recipe noted for you right now." became an approved memory). Pinned by `tests/test_memory_extractor_purity.py` (extractor contract + AST lockdown of the chat/voice call sites).
- Anaphoric capture: `memory_extractor.extract_and_ingest` keeps an in-process per-(user, session) last-USER-message LRU and passes it to `extract_candidates` as `prev_user_message`, anchoring entity-less corrections ("I meant saturday not friday") and pronoun-subject facts ("she's a doctor" after a person introduction) to the prior user turn. Fail-closed: no prior turn / no matching old value / gender-mismatched pronoun ‚Üí store NOTHING (never a guessed anchor). Only USER-authored text may enter the LRU (same purity contract as above). Pinned by `tests/test_memory_extractor_anaphora.py`.
- fact‚Üíperson linkage (Part A of `docs/architecture/gbrain-memory-graph-recall.md`): every producer keys a person-fact the SAME way as `person_extractor` ‚Äî resolved to a real contact ‚Üí `entity_type="person"`, `entity_id=<people.id>`; unresolved ‚Üí `entity_type="person_pending"`, `entity_id="slug:<name>"`. Never `person` + a bare name slug (mislabels an unlinked fact as graph-keyed). `memory_extractor.extract_and_ingest` resolves via `person_extractor._resolve_person_uuid` at ingest (opens a DB only when a person-fact is present). The idle re-linker `memory_digest._resolve_pending_person_links` runs in the dream cycle (`run_dreaming_cycle`) behind `ZOE_MEMORY_LINK_RESOLVER_ENABLED` (default OFF, true no-op): it re-scans `person_pending` facts, and once the contact exists rewrites `entity_id`‚Üí`people.id` + flips `person_pending`‚Üí`person` via a metadata-only `col.update` (no re-embed). Pinned by `tests/test_memory_linkage_hygiene.py`.
- Memory-maintenance loop observability (`routers/system.py` `_memory_digest_loop` nightly ~03:00 + `_memory_consolidation_loop` Sunday 04:00, started in `main.py`): both loops must stay observable so a broken pass can't fail silently again (the 2026-07 nightly-digest outage). On each completed run they call `memory_metrics.record_digest_run` / `record_consolidation_run` (via `_record_memory_loop`, which never raises) to persist a last-run timestamp + summed effect counts as Prometheus gauges (`zoe_memory_{digest,consolidation}_last_run_*`, on `/metrics`) and update the in-process `memory_loop_status()` state behind admin `GET /api/system/memory-loops/status` (age + `stale` flag ‚Äî digest grace ~26h, consolidation ~8d ‚Äî where a never-run or overdue loop reads stale). Run-status lines also land durably via `_attach_memory_loop_log_handler` (rotating `~/.zoe/zoe-data-memory-loops.log`, `ZOE_MEMORY_LOOP_LOG_PATH`) because the systemd unit doesn't reliably capture stdout. Keep both loops flag-gated (`MEMORY_DIGEST_ENABLED` / `MEMORY_CONSOLIDATION_ENABLED`, default on) and never let recording/logging break the loop body. Pinned by `tests/test_memory_loop_observability.py`.
- Tools register through the allow-list mechanism; every world-changing action goes through a proposal path.
- Background loops must not fork on the event loop thread. `asyncio.create_subprocess_exec` forks synchronously on the loop; in this large multi-threaded process a fork can deadlock pre-exec and freeze the whole API (live outage 2026-06-29: one wedged `hermes kanban` spawn made every endpoint, `/health` included, time out for days). The two safe patterns are centralised in `async_subprocess.py`: `run_to_completion` (spawn+communicate+timeout+kill entirely inside a thread pool via `subprocess.run`) and `spawn_pipe_process` (off-loop `subprocess.Popen` fork+exec, then wrap the already-open pipe fds in asyncio stream transports via `connect_read_pipe`/`connect_write_pipe`, which don't fork, so the rest of the worker keeps its async stdin/stdout interface). Prefer these for any new spawn ‚Äî a coroutine-side `wait_for` alone cannot fire when the loop thread itself is blocked, which is why the fork must happen off-loop. Run-to-completion sites: `executors/kanban_adapter.py`'s `_spawn_cli`, `tts_waterfall.py`'s `_spawn_tts_cli` (espeak-ng/ffmpeg TTS fallbacks), `pi_intent_classifier.py`'s print-transport governor (`run_to_completion`), and `intent_router.py`'s `_run_mcporter` (MCP fallback subprocess). Long-lived streamed sites: `pi_intent_classifier.py`'s Pi RPC governor worker (`async_subprocess.spawn_pipe_process`) and `zoe_core_client.py`'s Pi RPC brain subprocess (its own `_spawn_pi_process` ‚Äî a pre-existing specialised copy of the same logic; a follow-up may migrate it onto the shared helper).
- The bash tool runs commands via argv (`create_subprocess_exec`), never a shell. Outbound fetch/browser tools and panel proxies must route through `agent_safety`: HTTP fetches use `guarded_urlopen` (IP-pinned, env proxies ignored), Playwright pages must `await guard_browser_page(page)` before any `goto` (per-hop pre-connect validation), panel hosts go through `assert_panel_host`, and panel-display navigation (e.g. `panel_browser_screenshot`) goes through `assert_panel_url` (LAN-panel-only). Don't reintroduce raw `create_subprocess_shell`, unguarded `urlopen`, or a `page.goto` on a user URL without the route guard. Validating `page.url` only AFTER `goto` is insufficient ‚Äî the request was already sent.
- Every WebSocket endpoint in `main.py` (`/ws/voice/`, `/ws/push`, and the `/api/*/ws*` channels) must call `_enforce_ws_origin(websocket)` before accept/auth to block cross-site WebSocket hijacking. Allowed origins come from `_allowed_browser_origins()` ‚Äî the base `ALLOWED_ORIGINS` list plus `ZOE_ALLOWED_WS_ORIGINS` (comma-separated extra panel/kiosk hosts) ‚Äî which is the single source feeding BOTH this WS guard and the HTTP `CORSMiddleware`, so an operator-added origin can't be allowed for one and rejected by the other. Policy: a present-but-unlisted Origin is rejected (close 1008 / HTTP 403); a **missing** Origin is allowed (non-browser native kiosk/voice client ‚Äî browsers always send Origin on a WS handshake). Don't add a new WS endpoint without this guard.
- Known SSRF residuals (documented, accepted): (1) `guard_browser_page` validates each request URL in Python but cannot IP-pin Chromium's own resolver, so redirect-to-private is blocked pre-connect but exotic same-host DNS-rebinding (public at guard time, private at connect time) for the browser path is not closed ‚Äî full pinning is only available for `guarded_urlopen`. (2) `panel_browser_screenshot` constrains the initial `navigate_to` to a LAN panel host via `assert_panel_url`, but the navigation runs in the out-of-process Hermes broker, so an allowed LAN page that redirects to loopback/metadata inside the broker is only closeable broker-side. Closing either needs CDP `Fetch.requestPaused` resolved-IP inspection or a pinning proxy in the browser/broker.
- The stdio MCP server must derive the acting user from transport/session context, not tool arguments; non-admin tools may only target the acting user unless the tool is explicitly admin-authorized for cross-user action.
- Schema changes go through Alembic; never DROP/DELETE without WHERE and a backup.
