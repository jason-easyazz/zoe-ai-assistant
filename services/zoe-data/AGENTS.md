# services/zoe-data/ — production web + chat API

## Purpose

THE production Zoe API: FastAPI app served by host uvicorn on port 8000. Owns chat, intent routing, memory, MCP tools, auth integration, background jobs, and the OpenClaw/Hermes agent bridges.

## Ownership

- App entry and core: `main.py`, `database.py` (+ `db_pool.py`, `db_compat.py`, `alembic/` migrations), `auth.py`.
- Routing/NLU: `intent_router.py`, `intent_classifier_llm.py`, `pi_intent_classifier.py`, `routers/` (see child doc).
- SetFit router head + two-stage router: `semantic_router.py` optionally loads `models/router_head_logreg.joblib` (shadow) and, via `router_two_stage.py`, `models/router_head_mlp.joblib` (both trained in `labs/setfit-router`; never import from `labs/` at runtime). Flag `ZOE_ROUTER_HEAD` = `off` (default) | `shadow` (logreg head logs prediction/agreement, never routes) | `shadow2` (the FULL two-stage decision — MLP top-3 shortlist + 0.5 chat gate + grammar-constrained FunctionGemma sidecar on :11436 — computed in a background thread and logged, never routes) | `active` (the two-stage decision picks the domain; execution stays in expert dispatch; ANY sidecar failure/timeout/gate-abstain falls back to similarity → brain — the Gemma 4 brain remains the rock; this is a fast-tier front, proven 90.1%/0% chat-FP in `labs/router-90-campaign`). Sidecar unit: `scripts/setup/systemd/functiongemma-router.service`. Env: `ZOE_ROUTER_SIDECAR_URL`, `ZOE_ROUTER_TWO_STAGE_GATE` (0.5), `ZOE_ROUTER_TWO_STAGE_TIMEOUT_S` (1.5). Rollback = `ZOE_ROUTER_HEAD=off` + restart. Logs — utterance HASH only, never raw text — go to logger `zoe.router_head_shadow` and `data/router_head_shadow.jsonl` (gitignored); score with `scripts/maintenance/router_shadow_report.py`. The file is **rotated, never truncated**: append goes through `_append_shadow_line`, which rolls the log to `<log>.1`/`.2`/… past `ZOE_ROUTER_SHADOW_MAX_BYTES` (16 MB) keeping `ZOE_ROUTER_SHADOW_KEEP` (4) segments — ~80 MB ≈ 1.7 years at the observed ~133 KB/day (0 disables). Truncation is NOT an option: the self-train miner reads the WHOLE history (`--since` defaults to 0), so dropping records in place would silently starve the mine→label→ratchet loop. **Readers MUST go through `semantic_router.shadow_log_segments(path)`** (oldest→newest) — the miner and both shadow reports do; anything reading only `<log>` silently sees just the newest slice. Pinned by `tests/test_router_shadow_rotation.py`. The `scikit-learn`/`joblib` pins in `requirements.txt` must match the artifacts' training pins (`labs/setfit-router/requirements.txt`). Voice-path files ⇒ replay-gated. Pinned by `tests/test_router_head_shadow.py` + `tests/test_router_two_stage.py` (`ci_safe`). Public harness contract: `semantic_router.route_two_stage(text) -> RouterDecision` (dataclass: tool/args/confidence/source/latency_ms) — standalone with the sidecar up, no server; the 81-case corpus-through-prod-path harness ships separately against it.
- Pi intent/hybrid evidence JSONL: `pi_intent_evidence.py` — every write goes through `_append_jsonl`, which holds an **exclusive `fcntl.flock` on the target across BOTH the append and the `_prune_jsonl` trim**. The trim is a read-modify-write (read all rows → `seek(0)` → rewrite → `truncate`), and `record_pi_hybrid_production_evidence` always passes a positive `max_records`, so the trim runs on EVERY write. Unlocked, concurrent writers silently lose records — `pi_hybrid_production.try_pi_hybrid_production`'s fast-accept branch has two (the awaited accepted-decision write and the fire-and-forget `_schedule_pi_audit_after_fast_accept` audit write, both via `asyncio.to_thread`), and the lane is SCORED on this evidence. Keep the inode (never rename/replace the file — flock follows the open file description, not the path), and keep `_prune_jsonl` lock-free so it can't deadlock against its caller's hold. Pinned by `tests/test_pi_intent_evidence.py::test_append_jsonl_does_not_lose_records_under_concurrent_writers` (`ci_safe`).

ADR: `docs/adr/ADR-relationship-memory.md`. **Graph-adjacency recall boost** (`memory_service._semantic_search`, design `docs/architecture/gbrain-memory-graph-recall.md` Part B): a 7th additive blend term that lifts facts stored under people 1–2 hops from the person a recall query names (surfacing "my sister's husband's job" when the fact lives on the husband node). Gated behind BOTH `ZOE_RELATIONSHIP_GRAPH_ENABLED` AND a new sub-flag `ZOE_GRAPH_RECALL_BOOST` (default OFF); weight `ZOE_GRAPH_RECALL_WEIGHT` (default 0.30). Explicit user-triggered endpoint (`POST /api/people/{source_id}/merge-into/{target_id}`), owner-scoped, OFF the chat hot path (no voice replay gate). Roadmap item 4 of `docs/adr/ADR-relationship-memory.md`.

- First-turn-of-day greeting: `voice_greeting.py` — flag-gated (`ZOE_VOICE_GREETING_ENABLED`, default OFF, per-call env read) time-of-day greeting ("Good morning/afternoon/evening"). The **turn_stream** handler (`routers/voice_tts.py` `_wrapped`) emits it as its OWN leading spoken chunk (`{"chunk": -2, "provider": "greeting"}`) ahead of the answer — decoupled from the reply text (no UI-card pollution) and voice-only (NOT the shared `expert_dispatch`/chat core). - Segment-stitched TTS: `voice_stitch.py` — flag-gated (`ZOE_VOICE_STITCH_ENABLED`, default OFF) engine that renders high-frequency TEMPLATED voice replies (weather/time) by concatenating cached, finite-vocabulary phrase segments instead of synthesising the novel full sentence. `weather_segments(temp:int, cond, city)` / `time_segments(h,m)` build ordered natural-phrase segments from a bounded vocab (0–100, weather conditions, hours, minutes, am/pm); `concat_wavs` joins 24 kHz / mono / 16-bit WAV PCM with a 70 ms gap (refuses a format mismatch → caller falls back to full TTS); `stitch(segments, synth)` synths each (sidecar-cache-backed) + concats. After a one-time `prewarm_vocabulary` (wired flag-gated in `main.py` startup, paced + best-effort) every segment is a ~2 ms cache hit, so ANY weather/time value stitches in ~7 ms vs ~2 s synth. **Wired live:** `stitch_reply(text, synth)` strictly matches the exact intent replies (time `"It's 7:42 AM."`, weather `"It's 14 degrees and clear in {city}"` — both digit-form, converted to the word-form prewarmed segments), called in `routers/voice_tts.py` `_wrapped` (the dict-return path, before its per-sentence synth); any non-match / out-of-vocab / synth miss → None → unchanged full-sentence TTS (never emits guessed audio). Uses `ZOE_LOCATION_CITY`. Pinned by `tests/test_voice_stitch.py` (`ci_safe`).

- **Write-path DB exceptions log at WARNING+.** An `except Exception` guarding a SQL `INSERT`/`UPDATE`/`DELETE` must never end in bare `pass` or `logger.debug` — a swallowed write is a silent "Zoe forgot" with green health, the failure mode behind the FK trilogy (#1282) and the invisible pool leak (#1258). Log at `warning`/`error`/`exception` with enough context to diagnose (which write, which user/person, the exception). This is about VISIBILITY, not control flow: best-effort writers keep swallowing (a failed extraction must not break a turn) — they just say so. The Postgres→SQLite dialect-fallback `except` (the inner handler that retries the `?` param style) is dispatch, not a failure, and stays quiet. Enforced by `tests/test_write_path_exception_scan.py` (`ci_safe`, stdlib AST, sibling of `test_get_db_leak_scan.py`): modules join `WRITE_PATH_MODULES` as they are burned down; genuinely-harmless swallows need an `ALLOWLIST` entry with a one-line reason (today only `ui_layouts`, a self-healing compose-hint cache). `routers/voice_tts.py` is NOT yet listed — it is voice-runtime, so its handlers need the replay gate first.
- Tools register through the allow-list mechanism; every world-changing action goes through a proposal path.
- Background loops must not fork on the event loop thread. `asyncio.create_subprocess_exec` forks synchronously on the loop; in this large multi-threaded process a fork can deadlock pre-exec and freeze the whole API (live outage 2026-06-29: one wedged `hermes kanban` spawn made every endpoint, `/health` included, time out for days). The two safe patterns are centralised in `async_subprocess.py`: `run_to_completion` (spawn+communicate+timeout+kill entirely inside a thread pool via `subprocess.run`) and `spawn_pipe_process` (off-loop `subprocess.Popen` fork+exec, then wrap the already-open pipe fds in asyncio stream transports via `connect_read_pipe`/`connect_write_pipe`, which don't fork, so the rest of the worker keeps its async stdin/stdout interface). Prefer these for any new spawn — a coroutine-side `wait_for` alone cannot fire when the loop thread itself is blocked, which is why the fork must happen off-loop. Run-to-completion sites: `executors/kanban_adapter.py`'s `_spawn_cli`, `tts_waterfall.py`'s `_spawn_tts_cli` (espeak-ng/ffmpeg TTS fallbacks), `pi_intent_classifier.py`'s print-transport governor (`run_to_completion`), and `intent_router.py`'s `_run_mcporter` (MCP fallback subprocess). Long-lived streamed sites: `pi_intent_classifier.py`'s Pi RPC governor worker (`async_subprocess.spawn_pipe_process`) and `zoe_core_client.py`'s Pi RPC brain subprocess (its own `_spawn_pi_process` — a pre-existing specialised copy of the same logic; a follow-up may migrate it onto the shared helper).
- The bash tool runs commands via argv (`create_subprocess_exec`), never a shell. Outbound fetch/browser tools and panel proxies must route through `agent_safety`: HTTP fetches use `guarded_urlopen` (IP-pinned, env proxies ignored), Playwright pages must `await guard_browser_page(page)` before any `goto` (per-hop pre-connect validation), panel hosts go through `assert_panel_host`, and panel-display navigation (e.g. `panel_browser_screenshot`) goes through `assert_panel_url` (LAN-panel-only). Don't reintroduce raw `create_subprocess_shell`, unguarded `urlopen`, or a `page.goto` on a user URL without the route guard. Validating `page.url` only AFTER `goto` is insufficient — the request was already sent.
- Every WebSocket endpoint in `main.py` (`/ws/voice/`, `/ws/push`, and the `/api/*/ws*` channels) must call `_enforce_ws_origin(websocket)` before accept/auth to block cross-site WebSocket hijacking. Allowed origins come from `_allowed_browser_origins()` — the base `ALLOWED_ORIGINS` list plus `ZOE_ALLOWED_WS_ORIGINS` (comma-separated extra panel/kiosk hosts) — which is the single source feeding BOTH this WS guard and the HTTP `CORSMiddleware`, so an operator-added origin can't be allowed for one and rejected by the other. Don't add a new WS endpoint without this guard.
