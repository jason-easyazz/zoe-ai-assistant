# scripts/ — operational scripts

## Purpose

Operational scripting for Zoe hosts: setup, maintenance, deployment, migrations, testing helpers, and systemd unit templates.

## Ownership

- `setup/` — host/platform provisioning, including `setup/jetson/` systemd unit templates.
- `maintenance/` — recurring operational jobs: `deploy_zoe_data_when_ready.sh` (readiness/watch gate with no working-tree/live mutation, but it fetches remote-tracking refs; deploy mode pre-flights target `origin/main` content, builds a temporary pinned remote whose `main` is exactly that SHA, and invokes blessed `deploy_live.sh` with `origin` pointed at the pinned remote; after `deploy_live.sh` returns, the wrapper aborts if live `HEAD` differs from the pre-flighted SHA; post-deploy liveness checks remain informational because `deploy_live.sh` owns restart/rollback), `prune_worktrees.sh` (stale worktree cleanup), `greploop_guard.py` (Greptile fix-packet loop), `serena_mcp_capped.sh` (Serena MCP launcher — resolves the serena binary and self-provisions `jedi-language-server`, then wraps the server in a systemd user scope with `MemoryHigh`/`MemoryMax` caps, default 1G/2G via `SERENA_MEM_HIGH`/`SERENA_MEM_MAX`, so a runaway language server gets OOM-killed instead of exhausting the host; falls back to an uncapped launch if the user manager is unreachable. **`SERENA_NO_SCOPE=1` skips the scope** — required when launching under `serena-mcp.service`, whose unit carries the caps natively: nesting a scope inside a service reparents Serena to PID 1 and moves it into `app.slice`, escaping the service cgroup, which breaks `Restart=` tracking and makes the process look like an orphan to the reaper), `serena_mcp_health.sh` (read-only health check for the shared server — drives a real MCP `initialize` round-trip rather than just checking the socket, and FAILS if the port is bound beyond loopback; `[port]` arg defaults to 9121), `reap_stale_serena.py` (dry-run-first reaper for stale per-session Serena MCP servers — orphaned or multi-GB-swapped instances only, 30-min grace, kills require `--execute`; paired hourly user timer `zoe-serena-reaper.timer` in `setup/systemd/`. **Exempts the shared server** by cgroup match on `serena-mcp.service` — systemd owns its lifecycle, and an idle shared server drifts past the swap threshold, so without the exemption the hourly timer would reap the fleet's code-intel and discard its warm index), `zoe-memory-lint.py` (report-only memory Lint CLI — prints contradictions/stale/orphan/duplicate findings, never mutates stored memory), `compact_pipeline_state.py` (dry-run-first streaming compactor for the event-sourced pipeline state store — in-place rewrite under `LOCK_EX` with a retained backup; see Local Contracts for why compaction and not rotation), `remediate_ownerless_memories.py` (dry-run-first MemPalace ownerless-row audit/delete/backfill tool; destructive modes require `--execute` and create a whole-store directory backup before Chroma connects), triage generators, `pi_intent_probe.py` for local Pi/Gemma ambiguous-intent timing evidence, `router_shadow_report.py` (read-only summary of the SetFit router-head shadow log — agreement/coverage/disagreements/latency from `services/zoe-data/data/router_head_shadow.jsonl`), `router_shadow2_report.py` (read-only summary of the two-stage router shadow2 log — agreement / would-be fallback rate / chat-FP / would-be latency from `services/zoe-data/data/router_head_shadow.jsonl`; field-name tolerant), `router_rollout.sh` (operator driver for the staged two-stage-router rollout behind `ZOE_ROUTER_HEAD` — pre-flight, `--stage shadow2|active` with post-deploy probes, `--rollback`; auto-restores the previous flag value on any failure; runbook `docs/knowledge/two-stage-router-rollout.md`), and `music_discovery_batch.py` (weekly "Zoe Discovery" batch: runs the hidden digarr engine as an EPHEMERAL container — memory-gated ≥`ZOE_DISCOVERY_MIN_FREE_MB`, brain-idle-gated via llama-server `/slots`, always stopped+removed, exits non-zero on a leaked container — seeds one mood/discover from the local listening journal, then bridges the JSON into the "Zoe Discovery" MA playlist; scheduled from zoe-data behind `ZOE_MUSIC_DISCOVERY`, default OFF until a manual first run is verified), and `router_selftrain.py` (weekly router self-training loop: TRAIN → EVAL → RATCHET-PROMOTE → DEPLOY; retrains the FunctionGemma sidecar on mined real-traffic mistakes and promotes a candidate ONLY if every gate holds — no accuracy regression vs a LIVE-re-measured incumbent, zero chat false positives, p50 < 600 ms, and the voice replay gate actually *ran* and passed (the probe exits 0 on skip, so a skip is explicitly NOT a pass); independently re-verifies the candidate dataset never overlaps the frozen eval corpus; evaluates on a scratch sidecar so live :11436 is undisturbed; auto-rolls-back to last-known-good on any post-deploy failure; scheduled from zoe-data behind `ZOE_ROUTER_SELFTRAIN`, default OFF; runbook `docs/knowledge/router-selftrain-loop.md`), and `voice_gate_check.py` (the voice replay-gate HEARTBEAT check BOTH deploy paths invoke — the blessed `deploy_live.sh` and the continuous-deploy runner (`.github/workflows/deploy.yml`), each between the fetch and the tree-advance: "a gate that can silently not-run is not a gate", and CD is how changes actually reach the box. `voice_regression_probe.py` writes a durable result artifact on EVERY run — pass/fail/**skip**/**error** — at `~/.cache/zoe/voice_regression_last.json`; this checker reads it and, when the incoming git diff touches the voice runtime path (`voice_tts.py`/`zoe_core_client.py`/`fast_tiers.py`/`*kokoro*`/`*moonshine*`, override `ZOE_VOICE_GATE_PATHS`), REFUSES the deploy unless a FRESH (`<ZOE_VOICE_GATE_MAX_AGE_H`, default 24h) `status:pass` artifact matching the current baseline exists — a skip/missing/stale artifact is NOT a pass. It never runs the heavy Kokoro harness itself. Non-voice deploys are a no-op pass. Pinned by `tests/unit/test_voice_gate_check.py`; artifact contract in `docs/knowledge/voice-pipeline.md`).
- `perf/` — speed-regression harness: `measure_speed.py` (brain TTFT + gen tok/s probe vs the live llama-server), `measure_voice.py` (voice e2e latency + said-vs-did correctness, wrapping `services/zoe-data/tests/replay_samples.py`), and `measure_tts.py` (Kokoro TTS time-to-first-audio — times the live sidecar synth of the first speakable clause, reporting cache hit/miss; sources reply text from the replay corpus or a plain `--replies-file`). All are read-only and gated behind `ZOE_PERF=1`; see `perf/README.md` for the before/after workflow and the captured baseline.
- `lib/` — shared Python helpers imported by scripts across categories. `scripts/` is NOT a package: import a helper via the sibling-import convention (`sys.path.insert(0, str(Path(__file__).resolve().parents[1] / "lib"))`, then a plain module import), never a relative/package import. Currently `service_dir.py` — the ONE `--service-dir` resolution ladder shared by every voice-harness entrypoint (see Local Contracts).
- `deploy/`, `migrations/`, `testing/`, `train/`, `utilities/`, `preview/` — task-specific script groups.

## Local Contracts

- Scripts belong in a category subfolder, never in the repository root.
- **One service-dir ladder, in `lib/service_dir.py`.** Every voice-harness entrypoint (`maintenance/voice_regression_probe.py`, `perf/measure_voice.py`, `perf/measure_tts.py`) resolves the live `services/zoe-data/.env` through `resolve_service_dir()`: explicit `--service-dir` always wins → this repo's `services/zoe-data` if it has a `.env` → the MAIN worktree's (via git's `--git-common-dir`, never a hardcoded `/home/zoe/assistant`). So a run from a git worktree — direct or via the probe — needs no flag. Do not re-implement or fork the ladder per script; that drift is exactly what it exists to prevent (the probe invokes measure_voice as a SUBPROCESS, so the shared module also keeps the import graph acyclic). When no `.env` resolves anywhere it returns the in-tree default so the EXISTING loud skip/error still fires (`status=error`, exit 2): the ladder fixes the DEFAULT, never the failure mode — **a skip is not a pass**. Pinned by `tests/unit/test_probe_dsn_resolution.py`.
- `setup/kokoro_sidecar.py` persists its hot phrase cache under `~/.zoe/kokoro_cache/` (override `ZOE_KOKORO_CACHE_DIR`; disable with `ZOE_KOKORO_CACHE_PERSIST=0`) — `<sha256(key)>.wav` files plus `manifest.json` (hits/last_used/bytes), reloaded frequency-ranked on startup. Cache keys are VOICE-SCOPED (`<voice>|<lowercased text>`, `_phrase_cache_key`) so a voice switch (the panel's "Zoe's voice" setting or a `KOKORO_VOICE` change) never replays stale audio in the old voice — old text-only keys just miss and age out of the disk budget. Writes happen only off the synth hot path (periodic flush + shutdown) and are fail-open; disk budget via `ZOE_KOKORO_CACHE_MAX_DISK` / `ZOE_KOKORO_CACHE_MAX_DISK_BYTES`. Never point a temp/test instance at the live dir — use a `mktemp -d` dir and a non-10201 `KOKORO_SIDECAR_PORT`.
- Host install is orchestrated by `setup/install-jetson.sh` (idempotent: generates `.env` secrets → Docker spine → `deploy/migrate.sh` → `download_gguf_models.sh` → systemd spine → health gate) and touch panels by `setup/install-pi.sh` (SSH: wraps `touchscreen/install_touchscreen.sh` + `pi_voice_daemon_install.sh`). Shared bash helpers live in `setup/lib/common.sh` (logging, `gen_secret`, idempotent `env_set`) — `.gitignore` un-ignores `scripts/setup/lib/*.sh`. `download_gguf_models.sh` fetches the canonical Gemma 4 E4B-QAT + MTP rock (`unsloth/gemma-4-E4B-it-qat-GGUF`), NOT the old Llama/Qwen GGUFs.
- `setup/zoe_voice_daemon.py` ships WITH a companion module: `setup/zoe_voice_announce.py` (P-W2.3, stdlib-only) holds the announce poll/defer/expire decision logic so it is unit-testable in slim CI (`services/zoe-data/tests/test_voice_announce_daemon_logic.py` — the daemon itself imports pyaudio/numpy and can't load there). **Deploy BOTH files together** to `~/.zoe-voice/` on the panel (live path: `/home/pi/.zoe-voice/`, user service `zoe-voice`); the daemon degrades loudly (announce polling disabled, one WARNING) if the module is missing. The poll claims `GET /api/voice/announcements` with the existing `DEVICE_TOKEN`/`X-Device-Token` (no new provisioning), speaks through the same `/api/voice/speak` + `play_audio_b64` path as replies (barge-in, wake suppression, post-play cooldown all identical), never speaks while a turn/TTS/cooldown is live (defer; server TTL expires stale rows), and backs off quietly (up to 60 s) while zoe-data is down/restarting — a failed poll must never crash the daemon or touch the wake path. Env: `ZOE_ANNOUNCE_POLL_ENABLED` (default on), `ZOE_ANNOUNCE_POLL_S` (default 5).
- Installed systemd user units live in `~/.config/systemd/user/`; the copies here are templates. Keep both in sync when changing a unit.
- **The shared Serena MCP server (`setup/systemd/serena-mcp.service`, port 9121) is ONE process for the whole agent fleet**, replacing the old per-agent stdio spawn from `.mcp.json`. Three rules are load-bearing:
  - **Never use Serena's default port.** `--port` defaults to **8000**, which is zoe-data's PRODUCTION port — the default would collide with the live API. The port must always be set explicitly; 9121 is free, outside Zoe's dense 8000–8123 block, and below the ephemeral range (32768–60999) so the kernel cannot transiently steal it.
  - **Bind 127.0.0.1 only, never 0.0.0.0.** This server can read the entire repo; a LAN-exposed code-intel server is a real exposure. `serena_mcp_health.sh` fails on a non-loopback bind.
  - **The cap is now the FLEET bound, not a per-instance one.** The old stdio config capped each spawn at 1G/2G, which bounded each member but not the fleet (6 agents × 2G = 12G on a 15.6G box → the 2026-07-16 OOM). Keep `MemoryHigh`/`MemoryMax` on the unit; do not raise them to paper over a leak.
  Trade-off, measured and accepted: Serena runs ALL tool calls through one `SerenaAgent` TaskExecutor queue, so the shared server **serialises** the fleet (6 concurrent cold calls = 6.15s wall for 4.53s of work; every caller waits for the full drain). Serialised code-intel beats an OOMed voice brain. Do not "fix" this by going back to per-agent servers.
- Scripts run by timers/CI have no login session: prefix user-service systemctl calls with `XDG_RUNTIME_DIR=/run/user/$(id -u)`.
- Graphify is **retired and fully removed**: `graphify-out/` is not committed (gitignored), and the auto-refresh (`refresh_graphify.sh` + the `zoe-graphify-refresh` systemd timer/service), the repo write-back wrapper (`graphify_local_refresh.py`), and the former observe-only probe + shard tooling (`graphify_local_probe.py`, `graphify_local_shard_matrix.py`, `graphify_shard_sync_plan.py`, `graphify_shard_merge_validate.py`, `graphify_local_model_matrix.py`, `graphify_openrouter_cli.py`) have all been removed. Code intelligence is now provided by the codebase-memory MCP, re-indexed on demand. Do not re-introduce any graphify script, timer, or committed `graphify-out/` artifact.
- `prune_worktrees.sh` is dry-run by default; never pass `--execute` without operator review of the candidate list. Skips dirty, locked, live-checkout, unmerged, and recently-active worktrees.
- Destructive maintenance scripts must be dry-run by default, require explicit execution flags, and print the candidate list before deleting or rewriting anything.
- **`compact_pipeline_state.py` COMPACTS the pipeline state store; it must never ROTATE it.** `~/.zoe/engineering_pipeline_runs.jsonl` is event-sourced STATE, not append-history: `pipeline_store.save_state()` re-appends a full (growing) state snapshot every write, so the live store reached 1592 MB / 8421 records for 129 task_refs while the latest snapshot per ref totalled 1.3 MB. Rotation — correct for genuine append-history like `router_head_shadow.jsonl` (whose reports aggregate across lines) — would DELETE LIVE STATE here for any task_ref whose newest snapshot fell in the rotated-out segment. Do not bolt this onto the rotation mechanism. The tool's contract:
  - **Keep set = latest `state`-bearing record per `task_ref` UNION the last `--keep-tail` records.** Two readers, two needs: `pipeline_store._latest_state_from_lines` (last-match-wins) needs only latest-per-ref, but `engineering_harness_loop.parse_pipeline_findings` reads the last `DEFAULT_PIPELINE_TAIL` (200) lines as an EVENT STREAM and counts recurring `(task_ref, phase)` gate blocks — latest-per-ref alone collapses those counts to 1 and silently degrades triage. `--keep-tail` defaults to 200 to match; `tests/unit/test_compact_pipeline_state.py` pins the two constants equal. On the live store the union costs 14.2 MB vs 1.3 MB latest-only — still 99.11% reclaimed. Re-verify the reader set before narrowing the keep set.
  - **Rewrites IN PLACE under `fcntl.LOCK_EX`, never temp-file + rename.** flock is a property of the INODE: writers `open()` then flock, so a rename would leave an already-open writer appending to an orphaned inode — a silently lost state write with the lock excluding nothing. The inode must be preserved; the lock is held across the whole operation so writers block (correct) rather than race. The AST-level ban on `os.rename`/`os.replace`/`shutil.move` in the tool is pinned by test, as is the concurrent-writer survival property.
  - **Crash safety is by ordering, not atomicity:** compacted bytes are staged to a temp file and fsync'd, AND a full backup of the original is fsync'd, both BEFORE the original is touched; the backup path is printed first and retained for the operator to delete. A crash mid-rewrite is recoverable with a plain `cp` from the backup.
  - **Everything streams** — the store is never read into memory (peak RSS 21.6 MB scanning 1592 MB). That is the point: `load_latest_state` currently does `read().splitlines()` on the WHOLE file, which is the hazard being contained.
- `purge_orphaned_test_data.py` soft-deletes rows in the LIVE household DB, so its owner scope is a contract, not a heuristic: every owner pattern must be **fully anchored** and unable to match a real household id. Extend it only via `owner_pred()`/`TEST_OWNER_RE` — both are pinned by `tests/unit/test_purge_predicates.py`, which asserts the junk owners match AND that real ids (`jason`, `zoe-touch-pi`, …) never do. Verify any new pattern against the DB read-only before adding it; a past audit's assumed `test_calendar_*` rows never existed (zoe-data ignores the untokened `?user_id=` override and attributes such writes to `guest`). Note the `guest` arm is deliberately broad — it is the shared kiosk account, so a real panel-entered event is in scope; keep it dry-run-reviewed.
- Git history rewrites must run only in a disposable bare mirror clone with typed confirmation. Force-pushing rewritten history requires a separate explicit push flag, typed push confirmation, and exact confirmation of every resolved `origin` push URL. Pushing to multiple URLs is not atomic: every confirmed destination is attempted regardless of earlier failures, and the run exits non-zero with the full list of any destinations left divergent.
- Live deploy (`deploy_live.sh`) gates on TRACKED changes only, never on untracked files. Both the pre-pull clean-tree gate (`require_clean_tree`) and the pre-rollback guard (`require_no_tracked_dirt`) block only when uncommitted tracked worktree/index changes are present — a fast-forward or `git reset --hard` would clobber those. Untracked runtime artifacts on the live checkout (`data/chroma/`, `data/music-assistant/` sidecars, `homeassistant/custom_components/hacs/`, `services/zoe-data/data/music_discovery/`, …) must NOT block the deploy; a plain `git status --porcelain` counts them and made the blessed path refuse every normal deploy. Keep those runtime paths gitignored (root `.gitignore`) and keep the gates tracked-only. Pinned by `tests/unit/test_deploy_live_clean_tree.py`.
- `deploy_live.sh` and the continuous-deploy runner (`.github/workflows/deploy.yml`, push-to-`main` → fetch + `reset --hard` + restart) double-drive the SAME shared `/home/zoe/assistant` `.git`/worktree. They serialize on a shared **`flock /tmp/zoe-deploy.lock`** (script holds it across its whole mutating section; runner holds it across its pull/reset step) so they don't lose the ref-lock race. The GH `production` concurrency group only serializes runner-vs-runner. The runner's `reset --hard` intentionally has no clean-tree refuse — CD overwrites to match `main` by contract. The runner runs `voice_gate_check.py` after its fetch and BEFORE the reset (inside the same lock), so a voice-path change cannot auto-deploy without a fresh passing replay-gate artifact; it resets to the gate-checked SHA, not a re-read `FETCH_HEAD`. Fail-closed is intended — a blocked voice diff stays on `main` and blocks subsequent deploys until the probe is re-run (unwedge procedure: `docs/knowledge/merge-and-deploy.md`). Full CD mechanism: `docs/knowledge/merge-and-deploy.md`.
- Docker cleanup must not stop/remove containers or prune volumes; recurring cleanup is limited to dangling images and build cache.

## Forbidden

Rules for the autonomous loops that run from this subtree. `router_selftrain.py` promotes a model into the live voice path without a human in the loop; the ratchet is the only thing making that safe.

- **Never add a promotion bypass to `router_selftrain.py`** — no `--force-promote`, no "operator override", no env escape hatch. A candidate that cannot pass the ratchet is not safe to serve; fix the model, not the gate. `tests/unit/test_router_selftrain_gate.py` fails if a bypass symbol appears in the source.
- **Never promote** a router model that regresses overall accuracy against the live-re-measured incumbent, has chat false positives > 0, exceeds the p50 latency gate, or whose voice replay gate did not **run and pass**. A skipped probe (it exits 0 on skip) is NOT a pass.
- **Never train on the frozen eval corpus** (`labs/needle-benchmark/corpus.jsonl`). The candidate's held-out guard must be recorded *and* independently re-verified; abort otherwise.
- **Never delete the last-known-good GGUF** — rollback must always be possible (archive is copied, never moved; retention never drops below 1).
- **Never leave llama-server stopped**, and never leave the sidecar serving an unverified model.
- The loop's **only** permitted production mutation is swapping the sidecar model file and restarting `functiongemma-router.service`. It must never modify the live routing code path.

## Work Guidance

Log to journal (systemd) or an explicit log path; scripts that fail silently have already cost this repo a stale knowledge graph.

**`maintenance/zoe_ground_truth.sh` is the runtime-truth probe — the "what runs" companion to codebase-memory's "what exists".** Read-only and self-discovering (finds the zoe-data pid by port). **~1s on a healthy box, and bounded on failure** — parallel health checks, a single 2s DB-reachability preflight that gates every DB section, and per-call timeouts, so a broken dependency reports as "skipped/timeout" and never hangs the probe. Run it before trusting any doc's claim about what is live: it reads the brain lane from the running process env (not code defaults), the jobs that actually registered in `apscheduler_jobs`, Multica's own containers + DB, whether Hermes is live despite a "paused" doc, app-log health, and the deploy gap. Keep it strictly read-only and keep each check independent + non-fatal.

For `deploy_zoe_data_when_ready.sh`, run `pytest services/zoe-data/tests/test_deploy_zoe_data_when_ready.py`; the test uses fake git live trees and never restarts production.
